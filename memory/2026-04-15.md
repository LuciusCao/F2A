## 最终决策：JSON File 方案

### 决策理由（重新评估）

原始分析偏向 SQLite，但实现过程中发现：
1. **现有代码已准备 JSON 方案** - 已有 PersistedAgentRegistration 等类型定义
2. **尊重既有设计** - 避免推翻已有工作
3. **AgentRegistry 特点更适合 JSON**：
   - Agent 数量很少（几个到几十个）
   - 数据结构简单，无复杂查询需求
   - findByCapability 等查询在内存中执行，无需数据库索引
   - JSON 文件便于调试和查看
4. **实现更简单** - 无需额外的数据库连接管理

### SQLite vs JSON 对比回顾

| 维度 | SQLite | JSON File | 胜者 |
|------|--------|-----------|------|
| 实现复杂度 | 中等 | 简单 | JSON |
| 调试友好性 | 需工具 | 直接查看 | JSON |
| 查询性能 | 快（索引） | 慢（内存过滤） | SQLite |
| 数据完整性 | ACID | 无保证 | SQLite |
| AgentRegistry需求 | 不需要复杂查询 | **满足** | JSON |
| 现有代码准备 | 无 | **已有类型定义** | JSON |

### 结论
**AgentRegistry 数据规模小、查询简单、已有 JSON 准备 → JSON File 方案最优**

（SQLite 保留为 MessageStore 等复杂场景的方案）

## 方案调研（原始分析）

### SQLite 方案
**优势：**
- 项目已有成熟 SQLite 使用模式（message-store.ts），可复用经验
- better-sqlite3 已作为项目依赖，无需额外安装
- 支持复杂查询（如按能力查找 Agent）
- 数据完整性保证（ACID 特性）
- 适合中大规模数据
- 可添加索引优化查询性能

**劣势：**
- 需要处理数据库连接管理（生命周期）
- 需要设计表结构
- 稍复杂一些

### JSON File 方案
**优势：**
- 实现极其简单
- 易于调试和查看（直接打开文件）
- 无需额外依赖
- 备份/迁移方便（复制文件即可）

**劣势：**
- 每次修改需要全量写入（性能差）
- 无并发控制（文件锁）
- 无索引，查询效率低
- 数据损坏风险（写入中断）
- 不适合频繁写入场景

## 文件路径与格式

### 文件路径
- 文件名：`~/.f2a/agent-registry.json`
- dataDir 可配置（通过 AgentRegistryOptions）

### JSON 格式
```json
{
  "version": 1,
  "agents": [
    {
      "agentId": "agent:PeerIdPrefix:RandomSuffix",
      "name": "Agent名称",
      "capabilities": [{ "name": "capability-name", ... }],
      "peerId": "签发节点 PeerId",
      "signature": "签名Base64",
      "registeredAt": "2026-04-15T13:00:00Z",
      "lastActiveAt": "2026-04-15T13:05:00Z",
      "webhookUrl": "可选",
      "metadata": { 可选 }
    }
  ],
  "savedAt": "2026-04-15T13:05:00Z"
}
```

### 特殊处理
- `onMessage` 回调无法序列化，加载后不恢复
- `registeredAt` 和 `lastActiveAt` 为 ISO 8601 格式

## 实现要点

### 构造函数
```typescript
constructor(
  peerId: string,
  signFunction: (data: string) => string,
  options: AgentRegistryOptions = {}
)
```
- 添加 `dataDir` 和 `enablePersistence` 参数
- 自动调用 `load()` 加载持久化数据

### save() 方法
- 将内存 Map 转换为 JSON 格式
- 自动排除 `onMessage` 回调
- 使用 `JSON.stringify(data, null, 2)` 格式化输出
- 异步写入文件

### load() 方法
- 从 JSON 文件加载到内存 Map
- Date 字符串 → Date 对象转换
- 文件不存在时忽略（第一次运行）

### 自动保存触发点
- `register()` 后自动 save
- `unregister()` 后自动 save
- `updateName()` 后自动 save
- `cleanupInactive()` 清理后自动 save

### 调用方适配
- 方法签名保持不变（透明持久化）
- `register()` → 返回 `AgentRegistration`
- `unregister()` → 返回 `boolean`
- `updateName()` → 返回 `boolean`
- save/load 为内部方法，对调用方透明
