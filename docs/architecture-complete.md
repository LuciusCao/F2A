## 核心概念

### Node vs Agent

| 概念 | 职责 | 身份 | 生命周期 |
|------|------|------|----------|
| **Node** | 网络层：建立连接、传递消息、节点发现 | PeerID (libp2p) | 长期运行 |
| **Agent** | 业务层：执行任务、管理能力、积累信誉 | AgentID (独立) | 可迁移 |

**关键原则：**
- Node 是网络基础设施，不关心业务
- Agent 是业务实体，信誉绑定到 Agent 而不是 Node
- 一个 Node 可以服务多个 Agent

## 数据流示例

### 场景：Agent A 委托任务给 Agent B

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              任务委托流程                                    │
└─────────────────────────────────────────────────────────────────────────────┘

1. Agent A 决定委托任务
   ┌─────────────────┐
   │    Agent A      │
   │   (Node 1)      │
   │                 │
   │ 1. 选择目标 Agent (Agent B)                                           │
   │ 2. 检查 Agent B 的信誉 (通过 P2P 查询或本地缓存)                        │
   │ 3. 创建任务请求 (签名)                                                 │
   └────────┬────────┘
            │
            ▼
2. Node 1 发送消息
   ┌─────────────────┐
   │    Node 1       │
   │   PeerID: X     │
   │                 │
   │ 4. 查找 Agent B 所在的 Node (通过 DHT 或直接连接)                      │
   │ 5. 建立 P2P 连接 (如果尚未连接)                                        │
   │ 6. 发送消息: {to: "agent-bbb", from: "agent-aaa", task: {...}}        │
   └────────┬────────┘
            │
            │ P2P Network
            │ (libp2p stream)
            ▼
3. Node 2 接收消息
   ┌─────────────────┐
   │    Node 2       │
   │   PeerID: Y     │
   │                 │
   │ 7. 接收消息                                                              │
   │ 8. 解析目标 AgentID (agent-bbb)                                         │
   │ 9. 路由到本地 Agent B                                                    │
   └────────┬────────┘
            │
            ▼
4. Agent B 处理任务
   ┌─────────────────┐
   │    Agent B      │
   │   (Node 2)      │
   │                 │
   │ 10. 验证任务签名 (来自 Agent A)                                         │
   │ 11. 检查自己是否有能力执行                                              │
   │ 12. 执行任务                                                            │
   │ 13. 生成结果 (签名)                                                     │
   │ 14. 更新自己的信誉 (成功 +10)                                           │
   └────────┬────────┘
            │
            ▼
5. 结果返回
   ┌─────────────────┐
   │    Node 2       │ ──► Node 1 ──► Agent A
   │                 │
   │ 发送响应消息      │
   └─────────────────┘
            │
            ▼
6. Agent A 更新信誉
   ┌─────────────────┐
   │    Agent A      │
   │                 │
   │ 15. 验证结果签名                                                         │
   │ 16. 更新对 Agent B 的信誉评价 (如果需要)                                 │
   └─────────────────┘
```

## 身份与信誉模型

### 身份分离

```typescript
// Node 身份 (libp2p)
interface NodeIdentity {
  peerId: string;           // libp2p PeerID (如: 12D3KooW...)
  privateKey: Uint8Array;   // libp2p 私钥
  publicKey: Uint8Array;    // libp2p 公钥
}

// Agent 身份 (独立)
interface AgentIdentity {
  agentId: string;          // Agent 唯一标识 (如: agent-aaa-xxx)
  name: string;             // 显示名称
  privateKey: Uint8Array;   // Agent 签名私钥
  publicKey: Uint8Array;    // Agent 签名公钥
  createdAt: number;        // 创建时间
}

// Agent 能力
interface AgentCapabilities {
  skills: string[];         // ["code-generation", "file-operation"]
  computePower: number;     // 计算能力评分
  availability: number;     // 可用性 (0-1)
}

// Agent 信誉 (绑定到 AgentID)
interface AgentReputation {
  agentId: string;          // ⭐ 绑定到 Agent，不是 Node
  score: number;            // 0-100
  level: 'restricted' | 'novice' | 'participant' | 'contributor' | 'core';
  history: ReputationEvent[];
  lastUpdated: number;
}
```

## 迁移路径

> **⚠️ 修订**: 以下迁移路径已合并到 [RFC 005: F2A 架构统一](./rfcs/005-architecture-unification.md)

### Phase 1: 当前实现 (已完成)
- 1 Node = 1 Agent
- 信誉用 PeerID 作为键
- 功能可用，但架构不完整

### Phase 2: Agent 独立身份 (RFC 003)
- 引入 AgentId
- Agent 可以有自己的签名密钥
- 信誉键值从 PeerID 改为 AgentId

### Phase 3: 架构统一 (RFC 005)
- **MessageRouter/AgentRegistry 移入 F2A 核心类**
- 本地和远程消息走统一流程
- P2PNetwork 只负责传输

### Phase 4: 多 Agent 支持
- 一个 Node 可以注册多个 Agent
- 消息路由根据 AgentID 分发
- 每个 Agent 独立的信誉和能力

### Phase 5: Agent 迁移
- Agent 可以从 Node A 迁移到 Node B
- 信誉随 Agent 迁移
- 信誉证明 (签名验证)

**详细实施计划**: 见 [memory/2026-04-15.md](../memory/2026-04-15.md) "RFC 任务规划" 章节

## 总结

### Node 的职责
- ✅ 建立 P2P 连接
- ✅ 节点发现 (mDNS, DHT)
- ✅ 消息路由 (根据 AgentID)
- ❌ 不存储 Agent 信誉
- ❌ 不执行任务

### Agent 的职责
- ✅ 执行任务
- ✅ 管理能力
- ✅ 积累信誉 ⭐
- ✅ 签名验证

### 关键改进点
1. **身份分离**: Node 有 PeerID，Agent 有 AgentID
2. **信誉归属**: 信誉绑定到 AgentID，不是 PeerID
3. **多 Agent 支持**: 一个 Node 可以服务多个 Agent
4. **可迁移性**: Agent 可以在不同 Node 之间迁移，带信誉
